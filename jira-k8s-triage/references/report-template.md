# 分診報告範本

> 發到 Jira comment 的格式。語言跟隨原單語言（技術名詞保留英文），以下以中文示意。
> 範本使用 Jira 7.10 wiki markup：`h4.` 標題、`*粗體*`、`#` 編號清單、`{code}` 區塊。
> `<角括號>` 為填空位，發布前必須全部替換，不可殘留。

## 語氣與對外原則（約束所有範本）

- 平台團隊第一人稱、親切開場；把報案者當聰明的初學者，不是同行
- **正文禁止出現**：規則編號（PC/SR/PL）、內部術語（命中、分診、水位線、
  閘門、暫定分級）、以及我們驗證過程的描述
- 請報案者操作或補資料時：step-by-step 編號步驟，每步一個動作，
  寫到「在哪個畫面、點什麼、貼什麼給我們」的顆粒度
- 涉及 K8s 概念時附一句白話小知識（例：PVC 還有 Pod 掛載時是刪不掉的，
  這是 K8s 的保護機制）
- 結尾留互動口：「直接回覆這則留言」——回覆會由複診機制接手
- 分級結果反映在 Priority 欄位、接手人反映在 assignee 與 `[~提及]`；
  正文不複述內部分級代號（P2、S2×U1 這類不出現在正文）

## Footer 與規則編號列（所有範本共用）

每則 comment 一律以此作結：

```
----
🤖 由 AI triage 產生，經工程師審核後發布
<命中的規則編號，逗號相連，如 PC-001,PC-002；未命中任何規則則省略此行>
```

- footer 是複診水位線的辨識標記，**必須逐字一致、不可改寫**
- 編號列供我們追溯與統計；對報案者只是無害的 AI 代號，不解釋、不展開

## Something Broken 範本

```
h4. 👋 我們看過您回報的問題了
*發生了什麼*
<白話一兩句：什麼時候開始、影響範圍、關鍵錯誤（引用原文）>

*處理安排*
<誰接手：「這部分由 [~username] 接手處理」，或「需要請您這邊調整，步驟如下」；
時效用白話：「我們會在今天內處理」「已排入處理」>

*您現在可以做的*
# <立即可執行的動作；調查以 kube-dashboard 操作路徑為主，寫到點哪裡看哪裡>
<必要時附 YAML 修正片段（{code} 區塊）；涉及概念時加一句小知識>

*需要您幫忙補充*（如適用）
# <問題>（<為什麼需要：一句白話>）

有任何進展或問題，直接回覆這則留言，我們會再幫您看。
----
🤖 由 AI triage 產生，經工程師審核後發布
<規則編號列>
```

規則：
- 證據不足時「需要您幫忙補充」為必填段落，問題從 questioning.md 選出
  （只挑最能改變結論的 2-3 題，每題附一句白話原因）
- 止血指引以 kube-dashboard 為主（多數報案者無 kubectl 權限），
  kubectl 指令僅供有權限者替代

## Ask Platform 範本

```
h4. 👋 關於您的問題
*我們的理解*: <一句話重述問題>

*說明*
<白話解答，可 step-by-step；涉及 K8s 概念附一句小知識；
答不了時指出正確文件連結或負責團隊>

還有不清楚的地方，歡迎直接回覆這則留言。
----
🤖 由 AI triage 產生，經工程師審核後發布
<規則編號列>
```

規則：「建議沉澱為 FAQ」屬內部判定，寫入確認畫面的內部判定摘要，不進 comment。

## Pull Request Review 範本

```
h4. 🔍 PR 初步判讀
*PR*: <PR 連結>（<GitLab / Azure DevOps>）
*變更範圍*: 共 <N> 檔（<file1> +<a>/-<b>、<file2> +<c>/-<d>…）

*摘要*
<兩三句話：改了什麼、影響面>

*建議留意*
# <風險點與建議關注的檔案，白話說明為何要看；無則寫「未發現明顯風險」>

*後續*: 這個 PR 由 [~username] 接手 review；approve 由人工在 PR 平台執行。
----
🤖 由 AI triage 產生，經工程師審核後發布
<規則編號列>
```

規則：
- 「變更範圍」為必填欄位（Diff 證據閘門）——具體檔名與 +/- 行數只有
  實際讀取 diff 頁面才能取得；缺失或填不出具體數字時草稿無效，不得發布
- 「後續」的接手人依 routing.md「PR Review 分流」二擇一，為必填

## 請調整後重新提交範本（Policy 違規）

```
h4. 🙏 需要請您調整後重新提交
*情況說明*
<白話說明哪裡不符合平台的提交方式，引用具體證據（如 diff 中混合的
檔案清單）；不使用「違規」等指責性字眼>

*為什麼有這個規範*
<一兩句白話講規範的用意>

*請您這樣做*
# <step-by-step 拆分/修正步驟>

完成後直接回覆這則留言，我們會馬上接手處理。
----
🤖 由 AI triage 產生，經工程師審核後發布
<規則編號列>
```

規則：
- 不執行分流/轉診、不動 priority；發此 comment + 加 triaged label，
  並把 assignee 改指回 Reporter（單退回報案者手上）
- 「情況說明」必須引用具體證據（檔名、欄位、內容），不能只說不符合規範
- 報案者修正後回覆或重新提交，由複診（Follow-up）接手重新分診

## Follow-up 範本

```
h4. 🔁 收到您的回覆
*我們的回應*: <針對報案者提出的內容逐項回應，白話>

*更新後的處理*（有變動才列；無變動寫「原先的安排維持不變」）
<例：影響比原先評估的大，我們已調高處理優先順序，改由 [~username] 接手>

*下一步*
# <具體行動，一行一條>

----
🤖 由 AI triage 產生，經工程師審核後發布
<規則編號列>
```

規則：
- footer 必須保留——水位線機制依賴它辨識 AI 的最後一則 comment
- 只回應報案者實際提出的內容，不重複原報告已寫過的資訊
