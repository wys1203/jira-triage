# 分診報告範本

> 發到 Jira comment 的格式。語言跟隨原單語言（技術名詞保留英文），以下以中文示意。
> 範本使用 Jira 7.10 wiki markup：`h4.` 標題、`*粗體*`、`#` 編號清單。
> `<角括號>` 為填空位，發布前必須全部替換，不可殘留。

## Something Broken 範本

```
h4. 🏥 分診報告 (Triage Report)
*症狀摘要*
時間: <發生時間/持續時間>　地點: <cluster/namespace/資源>
關鍵症狀:
# <錯誤碼或現象，一行一條>

*分級*: <P2> (Severity: <S2> × Urgency: <U1>)
理由: <一句話>

*分流*: → <團隊名>
理由: <證據指向哪個領域>

*立即止血措施（非根治）*
# <可立即執行的動作；調查步驟以 kube-dashboard 操作路徑為主>
<必要時附建議的 YAML 修正片段，用 {code} 區塊包覆>

*需要補充的資訊*（如適用）
# <引導式提問>（原因: <一句話說明為何需要這個資訊>）

----
🤖 由 triage skill 產生，經工程師審核後發布
```

規則：
- 證據不足時「分級」標註 `暫定`，且「需要補充的資訊」為必填段落
- 「需要補充的資訊」的題目從 questioning.md 選出，只挑最能改變分級或分流結論的
  2-3 題；每題必附一句原因，報案者知道為什麼要提供才更願意回覆
- 止血措施的調查指引以 kube-dashboard 為主（多數報案者無 kubectl 權限），
  kubectl 指令僅供有權限者替代；可附 YAML 修正建議（{code} 區塊）
- 不含「升級觸發條件」段落——升級判斷由複診（Follow-up）依報案者回報的
  止血結果處理

## Ask Platform 範本

```
h4. 💬 平台諮詢回覆
*問題理解*: <一句話重述問題>

*解答 / 指引*
<直接給答案；或指出正確文件連結/負責團隊>

*FAQ 建議*: <若屬常見問題，註明「建議沉澱為 FAQ」；否則刪除此行>

----
🤖 由 triage skill 產生，經工程師審核後發布
```

規則：分級一律 S4，不含止血與升級條件段落。

## Pull Request Review 範本

```
h4. 🔍 PR 判讀報告
*PR*: <PR 連結>（<GitLab / Azure DevOps>）
*Diff 證據*: 共 <N> 檔變更（<file1> +<a>/-<b>、<file2> +<c>/-<d>…）

*改動摘要*
<兩三句話：改了什麼、範圍多大>

*風險點*
# <K8s 資源異動、權限變更、破壞性變更等，一行一條；無則寫「未發現明顯風險」>

*建議關注的檔案*
# <檔名：一句話說明為何要看>

*Review 建議*: <傾向可 approve / 需要討論>
理由: <一句話>

----
🤖 由 triage skill 產生，經工程師審核後發布；approve 由人工在 PR 平台執行
```

規則：「Diff 證據」為必填欄位——具體檔名與 +/- 行數只有實際讀取 diff 頁面
才能取得；缺失或填不出具體數字時草稿無效，不得進入收尾發布。

## Follow-up 範本

```
h4. 🔁 複診回覆 (Follow-up)
*回應摘要*: <一句話總結報案者的新回覆>

*更新後結論*（僅列有變動者；無變動則寫「原結論維持不變」）
<例：分級: S3 → S2（理由）；分流: → <新團隊>（理由）>

*下一步建議*
# <具體行動，一行一條>

----
🤖 由 triage skill 產生，經工程師審核後發布
```

規則：
- footer 必須保留——水位線機制依賴它辨識 skill 的最後一則 comment
- 只回應報案者實際提出的內容，不重複原報告已寫過的資訊
