# 收尾（四條路徑共用）

1. 報告草稿以原單語言撰寫（技術名詞保留英文），依 report-template.md 的
   「語氣與對外原則」行文；完整展示給使用者時，草稿上方附
   **內部判定摘要**——命中的規則編號、S×U 推導與 Priority 映射、
   轉診依據；此摘要僅供工程師審核與追溯，不隨 comment 發布
2. 以三選一的方式詢問使用者：**核可發布** / **修改後發** / **跳過此單**
   （環境有選項式提問工具就用，沒有就直接文字詢問並等待回覆）
   - 核可發布 → 草稿寫入暫存檔，**一次呼叫**：
     `jira-publish.sh <KEY> <暫存檔> [--priority <值>] [--assign <username>] [--transition <狀態>]`
     選項依情境取用：
     - `--priority`：有分級結論時帶（路徑 1、2；複診分級有變動也算），
       值依 routing.md「Jira Priority 映射」；PR Review 無分級不帶
     - `--assign`：分流/轉診對象為個人（`[~username]`）時帶 username；
       Policy 違規時帶 Reporter 帳號（jira-view 輸出 Reporter 括號內）；
       團隊分流或先天條件命中不帶
     - `--transition`：分流/轉診成立時（領域分流、SR、PR A/B）帶
       設定區的「分流後狀態」；先天條件命中、政策違規、純諮詢未分流不帶
     script 內建草稿 lint（footer 逐字、無殘留填空位、規則編號列格式與
     真實性），不過會拒發並列出原因——照原因修草稿重跑即可；
     「label 已存在略過」（複診）與「transition 失敗續行」皆屬預期，
     如實轉告使用者
   - 修改後發 → 依使用者指示改草稿，回到步驟 1 重新確認
   - 跳過此單 → 不留任何痕跡（不 comment、不加 label），下輪佇列仍會出現
3. 佇列模式 → 繼續下一張單
